# Recently 轻管理 AI 翻译 UI 设计

**日期：** 2026-04-15  
**范围：** `Blog` 前端仓库中的轻管理 `Recently` 模块  
**关联后端：** `V-IOLE-T/blog-mx-core` 已支持 `Recently` 的 AI 翻译生成、更新与读时回填

## 目标

在前端轻管理页面中，为 `Recently` 增加一个足够轻量、可直接操作的 AI 翻译入口，让站长不必切到完整后台，也能完成以下事情：

- 快速判断每条速记是否已有英文 / 日文翻译
- 对单条速记触发英文或日文的 AI 翻译生成
- 当已有翻译时，直接触发重新翻译更新
- 在操作完成后立即看到状态变化

这个 UI 的核心诉求不是“管理所有翻译能力”，而是让轻管理页面可以覆盖日常最常用的 `Recently` 翻译动作。

因此本次工作的重点不是再造页面，而是把轻管理现有速记卡片升级成“可见状态 + 可触发动作”的前端操作面。

## 用户已确认的设计偏好

### 交互形式

用户已明确选择：

- 不要在卡片上直接放多个语言按钮
- 每条速记只保留一个统一的 `AI 翻译` 入口
- 点击后再选择目标语言和动作

### 状态呈现

用户已明确选择：

- 翻译状态需要常驻显示在卡片上
- 例如直接显示 `EN 已翻译`、`JA 未翻译`
- 不接受“必须先点菜单才能看到状态”的隐藏式设计

## 方案对比

### 方案 1：行内状态 + 单一 `AI 翻译` 菜单（推荐）

- 每条速记卡片常驻显示英文、日文翻译状态
- 卡片操作区只新增一个 `AI 翻译` 按钮
- 点击按钮后弹出菜单，显示：
  - `生成英文` / `更新英文`
  - `生成日文` / `更新日文`

优点：

- 与轻管理页面定位最匹配
- 用户不需要记忆哪里看状态
- 操作入口统一，不会让卡片底部堆满按钮
- 后续若增加别的 AI 能力，也还有扩展空间

缺点：

- 每条卡片高度会略有增加

### 方案 2：顶部汇总状态 + 行内菜单

- 区块顶部显示整体翻译覆盖情况
- 单条卡片只保留 `AI 翻译` 菜单

优点：

- 适合从运营视角快速扫整体完成度

缺点：

- 对单条内容的即时判断帮助有限
- 实现成本更高，但不能显著提升核心操作效率

### 方案 3：行内状态 + 二级弹窗

- 点击 `AI 翻译` 后再弹一个 modal 选择语言与动作

优点：

- 扩展空间大

缺点：

- 对当前需求过重
- 会削弱轻管理“快速处理”的体验

## 选定方案

采用 **方案 1：行内状态 + 单一 `AI 翻译` 菜单**。

原因：

- 它最符合当前轻管理页面“快速查看、快速处理、不切走”的产品定位
- 能用最少的 UI 增量带来清晰的翻译可见性
- 不需要引入新页面或新的交互层级

## 详细设计

### 1. 卡片结构调整

在现有速记卡片中，保留：

- 正文内容
- 创建时间 / 编辑时间 / 赞踩数
- `编辑`
- `删除`

新增两类元素：

- 翻译状态徽标区
- `AI 翻译` 操作按钮

推荐结构顺序：

1. 速记正文
2. 元信息行
3. 翻译状态徽标行
4. 站长操作区：`编辑` / `删除` / `AI 翻译`

### 2. 翻译状态徽标

每条速记固定显示两个状态徽标：

- `EN 已翻译` 或 `EN 未翻译`
- `JA 已翻译` 或 `JA 未翻译`

状态规则：

- 若存在有效英文翻译，则英文显示 `已翻译`
- 若存在有效日文翻译，则日文显示 `已翻译`
- 否则显示 `未翻译`

视觉方向：

- `已翻译` 用较积极的浅色强调样式
- `未翻译` 用较弱的中性样式
- 保持与当前轻管理卡片的低干扰气质一致，不做过强色块

### 3. `AI 翻译` 菜单

卡片底部新增一个统一按钮：`AI 翻译`

点击后展示菜单项：

- `生成英文` 或 `更新英文`
- `生成日文` 或 `更新日文`

动态文案规则：

- 英文状态为未翻译时，显示 `生成英文`
- 英文状态为已翻译时，显示 `更新英文`
- 日文同理

这能让用户在不额外思考“现在该点生成还是更新”的情况下，直接看到系统建议动作。

### 4. 数据依赖与列表模型

当前轻管理 `RecentlySection` 只取最小字段用于展示。为了支持翻译状态，需要把列表数据升级到可识别翻译状态的版本。

前端至少需要在列表项中拿到：

- `id`
- `content`
- `created`
- `modified`
- `up`
- `down`
- 与翻译状态相关的元数据

推荐前端消费层逻辑：

- 不依赖单个 `isTranslated` 判断整体状态
- 直接基于 `availableTranslations` 或等价返回字段推导：
  - 是否存在 `en`
  - 是否存在 `ja`

原因：

- 一条速记可能同时存在多种目标语言翻译
- 轻管理页面关心的是“每种目标语言是否已覆盖”，而不是“当前返回项是否为翻译态”

如果当前列表接口还未直接返回足够字段，前端实现阶段应优先走最小增强：

- 优先复用现有 recently 列表接口
- 只追加轻管理所需的翻译元数据
- 避免为 UI 单独新建一个只服务前端的专属接口

### 5. 触发动作

`AI 翻译` 菜单项直接调用后端已存在的 `Recently` AI 翻译接口。

动作映射：

- 选择英文 -> 触发英文翻译生成 / 更新
- 选择日文 -> 触发日文翻译生成 / 更新

前端不区分“生成接口”和“更新接口”两套路由，只根据后端当前契约发起统一调用；“生成 / 更新”主要体现在 UI 文案层。

### 6. 操作反馈

菜单点击后需要有明确反馈：

- 按钮或菜单项进入 loading / disabled 状态
- 防止同一条速记重复点击导致重复提交

成功提示：

- `已提交英文翻译`
- `已提交日文翻译`
- 若是已有翻译时触发，可显示：
  - `已提交英文重翻译`
  - `已提交日文重翻译`

失败提示：

- 继续沿用现有 `toast.error(...)`
- 不额外引入 modal 级错误处理

完成后行为：

- 成功后刷新当前速记列表
- 刷新后重新计算状态徽标
- 用户无需手动刷新页面即可看到 `已翻译 / 未翻译` 变化

### 7. 首版边界

本次只做最小可用版本，不纳入以下能力：

- 翻译内容预览
- 批量翻译
- 多语言以外的更多 AI 操作
- 首页展示样式联动调整
- 单独的 AI 翻译管理页面

支持语言仅限：

- 英文 `en`
- 日文 `ja`

这样可以与当前站点语言切换范围保持一致，也避免轻管理页快速膨胀。

## 风险与约束

### 风险 1：前端当前列表模型字段不够

若现有 recently 列表接口没有返回多语言覆盖信息，前端无法准确判断 `EN / JA` 状态。

应对：

- 优先补齐轻管理所需最小字段
- 让状态推导建立在明确的翻译元数据上，而不是猜测

### 风险 2：翻译为异步任务，提交后不一定立刻完成

如果后端是异步入队，前端刷新时可能暂时还看不到 `已翻译`。

应对：

- 首版先接受“提交成功但状态稍后更新”的模型
- toast 文案强调“已提交”而不是“已完成”
- 若实测延迟明显，再考虑轮询或 socket 推送

### 风险 3：菜单 loading 粒度不对

若使用全局 loading，会导致所有卡片一起锁住。

应对：

- loading 状态按“单条速记 + 单个语言动作”维度管理

## 验收标准

以下条件同时满足即可认为设计达成目标：

1. 轻管理速记卡片上能直接看到 `EN` 与 `JA` 的翻译状态。
2. 每条速记只有一个统一的 `AI 翻译` 按钮，而不是多个散落按钮。
3. 菜单中的英文 / 日文动作会根据当前状态显示为“生成”或“更新”。
4. 点击菜单项后会调用后端 recently AI 翻译接口。
5. 成功后页面会刷新当前列表，并尽可能反映新的翻译状态。
6. 整个体验不需要跳到完整后台即可完成单条速记翻译管理。

## 实现建议

实现阶段建议优先走最小拆分：

1. 先补列表项类型与翻译状态推导
2. 再接入 `AI 翻译` mutation
3. 最后补状态徽标和按钮交互

这样能让验证更清晰，也有利于在接口字段不足时尽早发现阻塞点。
