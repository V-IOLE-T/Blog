# Vercel 前端部署收口设计

日期：2026-04-01

## 背景

当前 `Blog` 仓库已经完成 Yohaku 前端接管，线上 `https://418122.xyz` 也已经切到新的前端页面。

但最近几轮 handoff 和实测都说明了同一个问题：

- 当前 VPS 规格偏小
- 服务器现场直接执行 `docker compose build yohaku` 不稳定
- 曾经需要通过热更新运行中容器文件系统来让修复上线
- 运行中容器已经恢复功能，不代表镜像本身已经形成可重复部署产物

用户已经明确确认：这次收口的主线不是继续优化服务器现编，而是改成更稳定、可追踪、可回滚的正式前端发布路径。

## 目标

将当前前端生产发布方式从“VPS 本地构建并运行 `yohaku` 容器”收口为：

- `apps/web` 由 Vercel 托管部署
- `418122.xyz` 与 `www.418122.xyz` 切到 Vercel 前端
- `api.418122.xyz` 继续保留在当前 VPS，承载 MixSpace Core API
- 前端发布以 git 提交为准，不再依赖容器热覆盖或服务器现场编译

完成后应满足：

1. 仓库内存在可复用的 Vercel 部署配置与文档
2. GitHub Actions 能触发 Vercel 生产部署
3. 前端生产环境明确使用：
   - `NEXT_PUBLIC_API_URL=https://api.418122.xyz/api/v2`
   - `NEXT_PUBLIC_GATEWAY_URL=https://api.418122.xyz`
4. 生产前端部署可以和具体 git commit 对应
5. 当前线上前端修复不再依赖运行时热补丁作为长期手段

## 方案选择

### 方案 A：Vercel 托管前端，VPS 仅保留 API

做法：

- 将 `apps/web` 作为 Vercel 项目部署入口
- 通过 GitHub Actions 调用 Vercel CLI 完成生产部署
- 生产域名切到 Vercel
- VPS 继续提供 `api.418122.xyz`

优点：

- 直接绕开小规格 VPS 本地构建不稳的问题
- 部署、回滚、版本追踪都更标准
- 与当前仓库里已有 `apps/web/vercel.json` 和 Next.js 项目形态一致

缺点：

- 需要配置 Vercel 项目、环境变量和域名绑定
- 需要补一份清晰的切换与回退说明

### 方案 B：CI 构建镜像或 standalone，VPS 只拉产物

优点：

- 仍保留自托管前端
- 相比服务器本地 build 更可重复

缺点：

- 仍需维护镜像或制品分发链路
- 复杂度高于直接托管到 Vercel

### 方案 C：继续 VPS 本地构建，但流程脚本化

优点：

- 看起来改动最少

缺点：

- 没有解决 handoff 已经确认的根因
- 机器资源、磁盘和构建静默问题仍会反复出现

## 采用方案

采用方案 A：

- 前端主生产环境切换到 Vercel
- 后端 API 继续保留在现有 VPS
- 仓库中保留最小必要的部署自动化、环境变量说明、切换步骤和回退说明

## 设计范围

### 仓库内改动

本轮只做部署链路最小集：

- 新增或补齐 GitHub Actions 工作流，用于触发 Vercel 部署
- 新增部署说明文档，明确项目根目录、环境变量、域名绑定、发布与回退步骤
- 视需要补充根 `README.md` 中的生产部署推荐路径

### 默认不做

- 不改业务功能逻辑
- 不大改 `apps/web` 应用代码
- 不重做后端部署
- 不立刻删除 VPS 上现有 `yohaku` 容器，只把它降级为短期回退选项

只有在验证 Vercel 构建时发现明确阻塞时，才允许追加最小兼容性修补。

## 部署流程设计

目标流程如下：

1. 代码推送到目标分支
2. GitHub Actions 触发前端部署任务
3. 工作流使用 Vercel CLI 对 `apps/web` 发起生产部署
4. Vercel 使用预设环境变量完成构建
5. `418122.xyz` 与 `www.418122.xyz` 指向 Vercel 前端
6. 前端运行时继续请求 `https://api.418122.xyz/api/v2`
7. VPS 仅保留 API、数据库、对象存储等后端职责

## 环境变量策略

本轮优先明确前端生产必须项：

- `NEXT_PUBLIC_API_URL=https://api.418122.xyz/api/v2`
- `NEXT_PUBLIC_GATEWAY_URL=https://api.418122.xyz`

其余变量处理原则：

- 只把 Vercel 构建或运行确实需要的变量纳入文档
- 对敏感变量，优先通过 Vercel 项目环境变量注入，不写死在仓库
- 如果某些变量只对 VPS Docker 部署有效，应在文档中明确标记为“旧链路/回退链路使用”

## 风险与约束

1. 当前工作树不是干净状态，部署收口实现必须避免覆盖已有未提交业务改动
2. 仓库虽然已有 `apps/web/vercel.json`，但不代表整个项目已经具备完整的生产部署工作流
3. 域名切换后需要确认首页、owner 登录链路，以及“控制台”按钮依赖的 `config/url` 请求都能正常工作
4. 在正式确认稳定前，VPS 上现有 `yohaku` 容器不应立即清理，避免失去快速回退手段

## 验证方式

本轮至少验证：

1. 仓库内新增的部署工作流语法与配置可通过基础检查
2. 文档足够让冷启动接手者完成 Vercel 项目配置
3. 本地对 `@shiro/web` 的类型或构建验证仍通过，确保部署改动没有顺手破坏应用
4. handoff 中定义的关键线上能力被纳入验收：
   - 首页可访问
   - 前端可请求 `https://api.418122.xyz/api/v2`
   - owner 登录态下“控制台”按钮链路可继续工作
5. 最终交付中明确“何时可以下线 VPS 前端容器，何时只能作为回退保留”

## 非目标

本轮不承诺完成：

- 在这次改动里直接完成域名 DNS 切换
- 自动迁移 Vercel 项目和 Secrets
- 重写现有 Docker 部署体系
- 清理所有历史部署脚本
