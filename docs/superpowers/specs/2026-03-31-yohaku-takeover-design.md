# Yohaku 接管 Blog 仓库设计

日期：2026-03-31

## 背景

当前本地 `Blog` 仓库的旧博客业务文件已经全部删除，并保留了原有 git 历史与远端。

用户已明确确认：

- 当前 `Blog` 仓库继续沿用现有 git 历史与远端
- 前端改为私有仓库 `Innei-dev/Yohaku`
- 后端继续使用已存在的 `MixSpace Core`
- 服务器侧后续通过 SSH 直连处理部署与切流

## 目标

将当前 `Blog` 仓库改造成一个基于 `Innei-dev/Yohaku` 源码的新前端仓库，同时保留当前仓库的 git 历史和远端。

完成后应满足：

1. 当前仓库工作树主要内容变为 `Yohaku` 源码快照
2. 保留现有仓库的 git 历史和 `origin`
3. 明确记录上游来源与后续同步方式
4. 能确认本地构建入口与部署所需关键环境变量

## 方案选择

### 方案 A：直接导入 `Yohaku` 源码快照

做法：

- 只读拉取 `Innei-dev/Yohaku`
- 将其工作树内容复制进当前 `Blog` 仓库
- 保留当前 `Blog` 仓库的 git 历史

优点：

- 最快
- 与用户目标完全一致
- 后续改配置、提交和推送都在同一仓库完成

缺点：

- 不会保留 `Yohaku` 上游的 commit 历史，只保留代码快照

### 方案 B：用 subtree 或其他方式引入上游历史

优点：

- 能保留部分上游关系

缺点：

- 更重、更复杂
- 当前场景收益小

## 采用方案

采用方案 A：

- 当前 `Blog` 仓库保留现有历史和远端
- 工作树替换成 `Yohaku` 源码快照
- 额外补充一份上游说明

## 验证方式

本轮至少验证：

1. 私有仓库可读
2. `Yohaku` 的构建入口和必要环境变量已识别
3. 当前 `Blog` 仓库已被实际替换为 `Yohaku` 代码
4. git 历史和远端仍保持当前仓库身份

## 非目标

本轮不承诺完成：

- 服务器正式切流
- 线上构建和发布
- MixSpace 数据修复
- 最终域名验收
