# 首页左上角统一账号入口设计

日期：2026-04-02

## 背景

当前站点首页头部存在两套与账号相关的入口：

- 左上角头像入口：
  - 位于 `apps/web/src/components/layout/header/internal/AnimatedLogo.tsx`
  - 当前行为是单击回首页/直播、双击时 owner 进入控制台、访客进入登录页
- 右上角账号入口：
  - 位于 `apps/web/src/components/layout/header/internal/UserAuth.tsx`
  - 当前承担登录、显示账号头像、owner 菜单、登出等行为

用户希望首页前端更偏“访客视角”，不再把 owner 管理入口放在右上角，同时保留 owner 与访客都能从首页进入各自的账号流程。

用户已明确确认：

- 首页右上角不再显示当前账号头像入口
- 首页左上角头像改为唯一账号入口
- 未登录时左上角头像使用用户提供的 `enter.svg`
- 未登录时单击左上角头像，直接触发现有登录流程
- owner 登录后单击左上角头像，显示：
  - `进入控制台`
  - `轻管理`
  - `登出`
- 访客登录后单击左上角头像，显示：
  - 账号信息
  - 退出登录
- 登录后左上角应显示当前登录用户自己的头像

## 目标

将首页头部的账号交互统一收敛到左上角头像入口，形成一条清晰、可预测的单击链路，并去掉右上角重复的账号入口。

完成后应满足：

1. 首页左上角成为唯一账号入口
2. 未登录、访客登录、owner 登录三种状态下，头像与菜单行为清晰可区分
3. 右上角不再显示头像或管理菜单
4. owner 的 `控制台` 与 `轻管理` 入口仍可正常使用
5. 现有认证实现尽量复用，不重做认证体系

## 方案选择

### 方案 A：左上角单击统一入口，复用现有登录与菜单能力

做法：

- 将左上角入口从“单击/双击隐式行为”改为显式账号入口
- 未登录时单击直接触发现有 OAuth 登录弹层
- 登录后按 owner / 访客身份显示不同菜单
- 移除右上角重复头像入口

优点：

- 最符合用户当前目标
- 对现有认证逻辑改动最小
- 首页信息架构更清晰，只保留一个账号入口

缺点：

- 左上角入口的原始“回首页/直播”语义需要让位给账号语义

### 方案 B：左上角只做 owner 入口，访客仍保留右上角登录

优点：

- 对现有头部布局改动较小

缺点：

- 首页仍然存在两套账号入口
- 与用户“不要右上角头像”的目标冲突

## 采用方案

采用方案 A。

首页左上角头像入口调整为统一账号入口：

- 未登录：
  - 显示 `enter.svg`
  - 单击直接打开现有登录弹层
- 访客登录：
  - 显示访客头像
  - 单击打开账号菜单，内容为账号摘要和退出登录
- owner 登录：
  - 显示 owner 头像
  - 单击打开账号菜单，内容为：
    - `进入控制台`
    - `轻管理`
    - `登出`

同时移除首页右上角现有头像入口，避免重复交互。

## 交互设计

### 1. 左上角头像的视觉状态

- 未登录：
  - 使用用户提供的 `enter.svg` 作为头像资源
  - 鼠标悬浮时显示“登录”提示，明确这是可点击的登录入口
  - 实现时应将该 SVG 复制到仓库可控路径中使用，而不是直接依赖下载目录
- 访客登录：
  - 使用当前 session 的头像
- owner 登录：
  - 使用站点 owner 当前头像

### 2. 左上角头像的点击行为

- 未登录：
  - 单击直接触发 `useOauthLoginModal()` 对应的现有登录弹层
- 访客登录：
  - 单击打开菜单
  - 菜单展示：
    - 当前访客头像
    - 昵称
    - `@handle` 或 email
    - `退出登录`
- owner 登录：
  - 单击打开菜单
  - 菜单展示：
    - `进入控制台`
    - `轻管理`
    - `登出`

### 3. 右上角头像入口

- 不再显示当前 `UserAuth` 中的头像入口和菜单
- 若头部布局需要保留占位，应保留最小宽度稳定性，但不再承担账号交互

## 架构与组件边界

### 主要涉及组件

- `apps/web/src/components/layout/header/internal/AnimatedLogo.tsx`
  - 现有左上角头像入口
  - 将从“单击/双击行为”调整为统一账号入口
- `apps/web/src/components/layout/header/internal/UserAuth.tsx`
  - 现有右上角账号入口
  - 将移除首页上的头像/菜单职责，避免重复入口
- `apps/web/src/components/layout/header/Header.tsx`
  - 头部装配层
  - 需要确认右上角入口移除后的布局稳定性

### 状态来源

- owner 登录态：
  - `useIsOwnerLogged()`
- 访客 session：
  - `useSessionReader()`
- 登录动作：
  - `useOauthLoginModal()`
- owner 控制台地址：
  - `getAdminUrl()` 或现有等价能力
- owner 登出：
  - `apiClient.owner.logout()`
- 访客登出：
  - `authClient.signOut()`

### 资源接入

- `enter.svg` 需要在实现时复制到仓库内受版本控制的资源路径
- 不应直接从 `/Users/zhenghan/Downloads/enter.svg` 运行时读取

## 数据与状态流

### 未登录

1. 左上角渲染 `enter.svg`
2. 用户单击头像
3. 直接触发现有 OAuth 登录弹层
4. 登录成功后，相关 session 状态刷新
5. 左上角更新为当前登录用户头像

### 访客登录

1. 左上角读取 reader session 头像
2. 用户单击头像
3. 打开账号菜单
4. 用户可查看账号摘要或执行退出登录
5. 退出后恢复为 `enter.svg`

### owner 登录

1. 左上角读取 owner 头像
2. 用户单击头像
3. 打开 owner 菜单
4. 用户可打开：
   - 站外 `控制台`
   - 站内 `轻管理`
   - `登出`
5. 登出后恢复为 `enter.svg`

## 错误处理与兼容性

- 若访客 session 头像不存在：
  - 允许回退到统一占位样式，但不影响菜单功能
- 若 owner 控制台 URL 解析失败：
  - 菜单项不应报错卡死
  - 应保持点击无副作用或禁用该项
- 若 owner 与 reader 状态同时存在：
  - 以 owner 菜单优先，避免显示错误的访客菜单
- 若右上角入口移除后影响头部对齐：
  - 以保留布局稳定为优先，不强行压缩结构

4. 访客登录后左上角显示访客头像

   - `进入控制台`

9. `控制台` 仍可打开 `https://api.418122.xyz/proxy/qaqdmin#/dashboard`
