# MixSpace 占位数据初始化设计

**日期：** 2026-03-31

## 背景

当前线上 `MixSpace Core` 已可正常响应 `ping`，但 `aggregate/site` 返回“站点主人信息已丢失”，数据库中 `users/posts/notes/pages` 等核心集合为空，无法为 `Yohaku/Shiro` 提供最基本的站点数据。

用户已明确确认：

- 暂不迁移旧博客真实内容
- 先在线上写入一套最小可用的占位数据
- 目标是先让 `Yohaku/Shiro` 能跑起来，后续再人工替换内容

## 目标

将线上 `MixSpace Core` 从“空 Core”补齐为“可驱动 `Yohaku/Shiro` 的最小可用 Core”。

完成后应满足：

1. `https://api.418122.xyz/api/v2/aggregate/site` 不再报 owner 丢失
2. `https://api.418122.xyz/api/v2/aggregate?theme=shiro` 返回有效站点与主题聚合数据
3. 当根域名切换到 `Yohaku/Shiro` 时，首页至少能渲染基础站点信息与若干示例内容

## 范围

本次仅初始化最小必要数据，不追求完整站点还原。

包含：

- 1 个站点 owner 用户
- 站点基础配置
- Shiro/Yohaku 可消费的主题相关配置
- 1 个默认分类
- 1 到 3 篇已发布占位文章

不包含：

- 旧博客真实内容迁移
- 评论、草稿、统计数据恢复
- 历史附件、图片资源迁移
- 页面、说说、友链等非最小闭环能力，除非初始化过程强制要求

## 候选方案

### 方案 A：走系统已有的初始化/管理入口

优先查找线上现有 `MixSpace Core` 是否已经暴露：

- 初始化脚本
- 后台管理入口
- API 写入口
- 本地数据目录中的配置生成能力

优点：

- 更符合系统原始约束
- 数据结构更不容易写错
- 后续可重复使用

风险：

- 当前线上很可能没有现成管理面板
- 入口可能依赖额外环境变量或首次安装流程

### 方案 B：直接写 Mongo 最小数据集

如果系统不存在可用初始化入口，或入口无法在当前环境快速使用，则直接对 Mongo 写入最小集合数据。

优点：

- 最快可控
- 能直接对照接口缺失项补数据

风险：

- 需要先确认真实 schema
- 若写入字段不完整，接口仍可能报错

## 推荐方案

采用“**A 优先，B 兜底**”：

1. 先确认系统是否存在安全、可复用的初始化路径
2. 如果没有，再以数据库备份为前提直接写 Mongo
3. 写入后先验证 API，再考虑把根域名切到 `Yohaku/Shiro`

## 最小数据模型

### Owner

至少需要一个站点主人用户，字段以实际 schema 为准，但预期至少覆盖：

- 唯一标识
- 显示名
- 登录名或 slug
- 邮箱
- 头像
- 简介
- 角色/状态

### 站点配置

至少需要：

- 站点标题
- 站点描述
- 根域名
- 主题标识
- 基础 SEO/社交配置中的必要字段

### 内容

预期至少需要：

- 1 个分类
- 1 到 3 篇 `published` 状态文章
- 文章需带标题、slug、摘要、正文、发布时间、分类关系

正文采用占位中文内容，确保结构合法、页面可渲染即可。

## 数据生成原则

- 内容必须显式标记为占位数据，方便后续人工识别和替换
- 文章标题和摘要用自然中文，避免全是测试字符串
- 正文尽量使用 Markdown 兼容结构
- 所有时间字段统一使用当前时间附近值，减少排序异常
- 不预设复杂关系，避免引入非必要约束

## 验证方式

写入完成后至少验证：

1. `ping` 仍然正常
2. `aggregate/site` 返回 owner
3. `aggregate?theme=shiro` 返回主题聚合结果
4. 若接口中提供 posts 列表或首页聚合信息，应能看到占位文章

如接口层验证通过，再进入前端切流验证。

## 风险与回滚

主要风险：

- schema 与推测不符，导致接口继续报错
- 初始化顺序要求高，单写用户或文章不足以恢复聚合接口
- 主题配置字段缺失，导致 Shiro 聚合仍为空

回滚策略：

1. 写入前导出目标数据库相关集合
2. 所有新增记录使用可识别前缀，便于定向删除
3. 若直接写库失败且影响接口，优先回滚新增记录，不改动现有服务配置

## 结论

本次先不做真实内容迁移，只以“让 `Yohaku/Shiro` 吃到最小可用数据”为目标。

执行顺序为：

1. 核实线上 `MixSpace Core` 的真实初始化入口或 schema
2. 备份数据库
3. 写入最小占位数据
4. 验证聚合接口
5. 再进入前端切流
